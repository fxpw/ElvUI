# План рефакторинга Nameplates на новый API (ElvUI + ElvUI_OptionsUI)

## 1) Цель

Основная цель: перенести конфигурацию и рантайм-модуль Nameplates из retail-источника в текущую кодовую базу Sirus с адаптацией под доступный игровой API и архитектуру проекта.

В рамках этого плана:

- Заменить содержимое настроек в ElvUI_OptionsUI\Nameplates.lua на эквивалент из D:\repos\ElvUI_retail\ElvUI_Options\Core\Nameplates.lua с адаптацией под локальный формат опций.
- Перенести код модуля из D:\repos\ElvUI_retail\ElvUI\Core\Modules\Nameplates\* в ElvUI\Modules\Nameplates\* со всеми нужными зависимостями и адаптацией API.
- Использовать ElvUI\interface_code\* как эталон поддерживаемого API клиента.

Вне рамок (можно вынести в отдельные задачи):

- Полный редизайн не связанных с Nameplates модулей ElvUI.
- Прочие UI-системы, если они не являются жесткой зависимостью Nameplates.

## 3) Текущая дельта (уже выявлено)

### 3.1 Расхождение слоя опций

- Текущий ElvUI_OptionsUI\Nameplates.lua построен на legacy-формате таблиц опций.
- Retail-файл опций опирается на ACH-абстракции (E.Libs.ACH) и более новую структуру групп/ключей.

Вывод:

- Прямой copy-paste небезопасен: нужна явная трансформация в локальный стиль.

### 3.2 Расхождение структуры модуля

В текущем ElvUI\Modules\Nameplates есть локальные файлы и структура, которые отличаются от retail-набора.

В retail-наборе присутствуют другие блоки Elements/Plugins (например: BossMods, ClassPower, Power, Threat, RaidTargetIndicator, Tags и т.д.).

Вывод:

- Перенос нельзя делать простым перезаписыванием папки; нужна карта файлов и функциональная карта.

### 3.3 Риск несовпадения API

- Доступный API клиента описан в interface_code\C_NamePlates.lua.
- Retail-код может использовать отличающиеся события, CVars и сигнатуры функций.

Вывод:

- Нужен отдельный слой API-адаптации и проверка совместимости по каждому критичному вызову.

## 4) Стратегия рефакторинга (по фазам)

## Фаза 0. Подготовка и безопасность

1. Использовать текущую миграционную ветку: new_plates.
2. Снять снимки текущих файлов:
   - ElvUI_OptionsUI\Nameplates.lua
   - ElvUI\Modules\Nameplates\*
3. Зафиксировать базовое поведение в игре:
   - видимость friendly/enemy неймплейтов
   - кастбар
   - ауры
   - реакции StyleFilter
4. Зафиксировать scope первого прохода: только функциональный паритет без новых фич.

Результат фазы:

- Безопасная точка отката и базовый чеклист поведения.

## Фаза 1. Инвентаризация зависимостей и карта миграции

1. Построить граф зависимостей retail Nameplates:
   - внутренние зависимости внутри папки Nameplates
   - внешние зависимости (Tags, Cooldowns, Filters, SharedMedia и т.п.)
2. Составить карту используемых символов/API:
   - WoW globals
   - C_NamePlate.*
   - хелперы движка ElvUI (E, NP, конфиг-хелперы)
3. Классифицировать каждую зависимость:
   - есть без изменений
   - есть, но отличается сигнатура/контракт
   - отсутствует (надо портировать или шимирировать)

Результат фазы:

- Матрица зависимостей Nameplates.

## Фаза 2. Миграция опций (ElvUI_OptionsUI\Nameplates.lua)

Требование фазы:

- Набор настроек как в retail, но в рабочем локальном формате опций.

Шаги:

1. Извлечь из retail все группы опций и ключи.
2. Построить таблицу соответствий ключей:
   - retail key -> локальный путь в БД (E.db / E.private / E.global)
   - источник дефолтов (P / V / G)
3. Преобразовать ACH-конструкции в локальную схему опций:
   - Group, Toggle, Range, Select, Input, Execute, MultiSelect
   - динамические disabled/hidden/get/set
4. В первую очередь перенести сложные секции:
   - unit settings
   - buff/debuff filters + priority
   - castbar text/icon/font
   - style filter triggers/actions
5. Проверить соответствие путей навигации (SelectGroup) фактическому дереву опций в проекте.
6. Временные совместимые обертки использовать только при необходимости.

Критерии приемки:

- Новый блок Nameplates открывается без Lua-ошибок.
- Критичные группы корректно читают и сохраняют значения.
- Нет ссылок на отсутствующие либы/хелперы.

## Фаза 3. Миграция ядра модуля (ElvUI\Modules\Nameplates)

Шаги:

1. Сначала выровнять порядок загрузки:
   - синхронизировать Load_Nameplates.xml с выбранным набором файлов
   - обеспечить корректный порядок: ядро -> элементы -> плагины
2. Перенести базовые файлы:
   - Nameplates.lua
   - StyleFilter.lua
   - общие utility-блоки
3. Переносить Elements и Plugins по приоритету зависимостей:
   - сначала базовая функциональность (видимость, здоровье, каст)
   - затем вторичные визуальные расширения
4. Для каждого перенесенного файла:
   - закрыть API-расхождения
   - сохранить необходимое Sirus-специфичное поведение
   - не допускать побочных эффектов за пределами Nameplates
5. Все решения keep/replace фиксировать в заметках миграции.

Критерии приемки:

- Неймплейты корректно создаются и удаляются.
- Работают таргетинг, уровни фреймов, ауры, кастбары и фильтры.
- Нет критичных рантайм-ошибок.

## Фаза 4. Адаптационный слой под новый API (по interface_code)

1. Сделать матрицу совместимости по вызовам C_NamePlate в мигрированном коде:
   - поддерживается напрямую
   - требует адаптации параметров
   - отсутствует (нужен fallback)
2. Ввести тонкий адаптер внутри модуля Nameplates для изоляции клиентских различий.
3. Пропустить обработку событий через адаптер там, где отличается payload.
4. Нормализовать работу с CVar:
   - проверить все CVar, используемые retail-кодом
   - валидировать их наличие на клиенте
   - добавить безопасные проверки для отсутствующих CVar

Критерии приемки:

- Нет hard-fail вызовов к неподдерживаемому API.
- Логика синхронизации CVar стабильна после reload/login.

## Фаза 5. Совместимость данных и дефолтов

1. Добавить миграцию существующих профилей:
   - сохранять старые значения, где это возможно
   - инициализировать новые ключи безопасными дефолтами
2. Проверить соответствие схем P.nameplates, V.nameplates, G.nameplates мигрированным опциям.
3. Добавить version gate для одноразовой миграции профиля.

Критерии приемки:

- Апгрейд существующих профилей проходит без разрушения настроек.
- Чистая установка и обновленный профиль работают одинаково стабильно.

## Фаза 6. Валидация и контроль регрессий

### Функциональный чеклист

- Friendly/enemy/player nameplates
- Масштаб, альфа, правила видимости
- Castbar (interrupt source, icon, time format)
- Ауры (якоря, рост, сортировка)
- StyleFilter (triggers/actions)
- Target/highlight/threat
- Totem/unique unit логика (если включена)

### Технический чеклист

- Нет Lua-ошибок при загрузке, в бою, на арене/бг, в данжах/рейдах.
- Нет nil-доступов в callback-ах опций (get/set/disabled/hidden).
- Нет нового taint в click/secure сценариях.

### Чеклист производительности

- Контроль пиков CPU на NAME_PLATE_UNIT_ADDED/REMOVED.
- Аудит повторных полных reconfigure-вызовов.
- Контроль отсутствия утечек и бесконтрольного роста таблиц кэша.

## 5) Риски и меры снижения

1. Риск: ACH-зависимые retail-опции не переносятся напрямую в legacy-формат.
   - Мера: явная трансформация по группам и staged-конвертация.
2. Риск: расхождение событий/API между retail и текущим клиентом.
   - Мера: адаптер + runtime-guards + fallback-логика.
3. Риск: скрытые зависимости от других современных подсистем ElvUI.
   - Мера: матрица зависимостей до массовой замены файлов.
4. Риск: поломка локальной Sirus-специфичной логики.
   - Мера: список локальных патчей и осознанное повторное применение после базового переноса.

1. ElvUI_OptionsUI\Nameplates.lua отражает retail-набор настроек и работает в локальном UI-формате.

4. В реальных игровых сценариях нет блокирующих Lua-ошибок.
5. Существующие пользовательские профили корректно обновляются.
